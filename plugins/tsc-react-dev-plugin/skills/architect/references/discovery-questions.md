# Discovery Questions

Use these questions during the requirements discovery phase. Select questions relevant to the user's context - do not ask all of them. Ask 2-4 questions at a time to avoid overwhelming the user.

## Problem Understanding

- What problem are you trying to solve?
- Who are the users affected by this?
- What do users currently do to work around this?
- What happens if we don't build this?
- How will you measure success?

## Scope and Boundaries

- What is the minimum viable version of this feature?
- What is explicitly out of scope?
- Are there related features or upcoming work this needs to accommodate?
- What are the hard deadlines or constraints?

## User Experience

- Do you have mockups, wireframes, or design references?
- What are the key user flows?
- Are there existing patterns in the app we should follow?
- What accessibility requirements apply?
- What devices and browsers must be supported?

## Data and Integration

- Where does the data come from?
- Are the APIs already built, or do they need to be designed?
- What is the expected data volume / scale?
- Are there real-time requirements (WebSocket, polling, SSE)?
- What happens when the API is slow or unavailable?

## Technical Context

- What is the current tech stack and are there constraints on library choices?
- Are there existing components or patterns we should reuse?
- Are there performance budgets or targets?
- How is state currently managed in this area of the app?
- Are there testing requirements or coverage targets?

## Visual Design

- What is the brand personality or visual tone? (e.g., professional, playful, minimal, bold)
- Are there existing brand guidelines, color palettes, or design tokens?
- Are there reference sites or apps whose visual style you admire?
- Is there an existing design system or component library in use?
- What level of visual fidelity do you need? (wireframe, low-fi mockup, high-fi mockup)
- Are there specific accessibility requirements for color contrast or motion?
- Should the design follow an existing visual language or establish a new one?

## Team and Process

- Who needs to review or approve this design?
- Are there other teams whose work this depends on or affects?
- Is this greenfield or does it need to integrate with existing code?
- What documentation does your team expect before development starts?
